**状态：** 草案

**作者：** Claw-to-Claw Protocol 贡献者

## 1. 概述

Claw-to-Claw (C2C) 协议是一个**受治理的 AI 技能分发协议**。它定义了 AI 智能体之间如何发现、请求、验证和学习技能 — 在每一步都有人类和组织的监督。

C2C 回答了一个现有协议都没有涉及的问题：

> **"这个智能体是否被允许学习这个技能？"**

与运行时协议不同（MCP 用于工具调用，A2A 用于智能体通信，ACP 用于 IDE-智能体交互），C2C 关注的是**永久性知识传授** — 一个智能体教会另一个智能体它之前不具备的能力。

### 1.1 协议栈

```
┌─────────────────────────────────────────────────────┐
│              C2C 协议栈                               │
│                                                      │
│  第4层：治理（v4.0 规划中）                            │
│  └─ 策略、自动审批规则、组织管理                        │
│                                                      │
│  第3层：身份与信任（v3.0 规划中）                       │
│  └─ 所有者身份、技能认证、组织验证                      │
│                                                      │
│  第2层：知识瓶子格式（v2.0 规划中）                     │
│  └─ bottle.json 清单、多表示形式、JSON Schema 接口     │
│                                                      │
│  第1层：传输（v1.x 当前）                              │
│  └─ 人类中继（Lobster Distill）                       │
│  └─ 直接下载（c2cprotocol.org，规划中）                │
│                                                      │
│  第0层：加密（v1.x 当前）                              │
│  └─ AES-256-CBC、一次性密码、24小时过期                 │
└─────────────────────────────────────────────────────┘
```

### 1.2 设计目标

- **受治理的分发**：技能默认私有；传授需要所有者同意
- **跨框架**：技能可在 AI 框架间移植（OpenClaw、LangChain、AutoGen、Dify 等）
- **人类参与审核**：人类控制所有传输，维持监督和信任
- **传输无关**：支持任何信道 — IM、邮件、API 或人类中继
- **隐私优先**：端到端加密，一次性密码
- **零依赖**（第1层）：仅使用普遍存在的系统工具
- **渐进式采用**：每一层独立提供价值

### 1.3 非目标

- 实时智能体间通信（参见 [A2A](https://github.com/a2aproject/A2A)）
- 工具发现和运行时调用（参见 [MCP](https://github.com/modelcontextprotocol)）
- IDE-智能体会话管理（参见 [ACP](https://agentclientprotocol.com/)）
- 替代现有包注册中心（ClawHub、npm、pip）

### 1.4 C2C 与其他协议的关系

| 协议 | 用途 | 层级 | 与 C2C 的关系 |
|------|------|------|---------------|
| **MCP** | 工具调用 | 执行层 | 学到的技能可注册为 MCP 工具 |
| **A2A** | 智能体通信 | 发现层 | 智能体可通过 A2A 发现技能，然后通过 C2C 传授 |
| **ACP** | IDE-智能体交互 | 会话层 | 正交 — 不同使用场景 |
| **ClawHub** | 技能仓库（公开） | 分发层 | 互补 — C2C 增加私有、受治理的分发 |

```
发现层：  A2A    — "你会做什么？"
学习层：  C2C    — "教我也会。"
执行层：  MCP    — "帮我做这件事。"
```

### 1.5 C2C 与包注册中心（ClawHub、npm、pip）的区别

| | 包注册中心 | C2C（c2cprotocol.org） |
|---|---|---|
| **模型** | 仓库（文件托管在平台上） | 目录（仅存元数据；技能留在所有者处） |
| **隐私** | 默认公开 | 默认私有 |
| **交付** | 任何人可下载 | 所有者必须同意每次传输 |
| **范围** | 特定框架 | 跨框架 |
| **目标** | 开源社区 | 个人 + 企业 + 组织 |
| **类比** | npm / Docker Hub | DNS / Amazon 商城 |

**两者互补，非竞争。** 一个技能可以同时在 ClawHub 上公开，并在 c2cprotocol.org 上注册以实现受治理的跨框架分发。

## 2. 核心概念 — 知识瓶子（Knowledge Bottle）

C2C 协议围绕两大核心机制展开：**知识蒸馏** 与 **人类中继**。参考实现名为 **Lobster Distill**。

### 2.1 第一步：技能蒸馏

源 AI 将执行某项任务所需的内部知识 — 包括提示词、工具调用逻辑、参数约束等 — 提纯为一个紧凑的、结构化的数据包，称为 **知识瓶子（Knowledge Bottle）**。

### 2.2 第二步：加密与封装

知识瓶子使用 AES-256-CBC + PBKDF2 和一次性密码进行加密。只有持有正确密钥的接收方才能解开。

### 2.3 第三步：传输

加密后的瓶子交付给接收方。C2C 支持多种传输机制：

| 机制 | 描述 | 状态 |
|------|------|------|
| **人类中继** | 人类复制 Notes 并通过任意文字信道转发 | v1.0（当前） |
| **直接下载** | 接收方经所有者同意后从 c2cprotocol.org 下载 | v3.0（规划中） |
| **API 传输** | 授权智能体间的程序化传输 | v4.0（规划中） |

### 2.4 第四步：目标 AI 加载与执行

目标 AI 解密知识瓶子，解析技能描述，动态加载技能 — **立即获得** 与源 AI 相同的能力。

```
源 AI             知识瓶子              人类中继           目标 AI
  │                  │                    │                  │
  │ 1. 蒸馏 ──────→ │                    │                  │
  │                  │ 2. 加密 ────→     │                  │
  │                  │    上传临时存储    │                  │
  │                  │ 3. 生成 Notes ──→ │                  │
  │                  │                    │ 4. 转发 ──────→ │
  │                  │                    │                  │ 5. 下载
  │                  │                    │                  │ 6. 解密
  │                  │                    │                  │ 7. 安装
  │                  │                    │                  │ 8. 技能激活 ✅
```

## 3. 术语

| 术语 | 定义 |
|------|------|
| **智能体（Agent）** | 能够执行技能的 AI 系统（如 OpenClaw 实例） |
| **技能（Skill）** | 可传授的可执行知识单元（代码、配置、文档） |
| **知识瓶子（Knowledge Bottle）** | AI 技能经蒸馏、加密后的传输包 — C2C 的核心传输工件 |
| **发送方（Sender）** | 打包并加密技能以供传授的智能体 |
| **接收方（Receiver）** | 下载、解密并安装技能的智能体 |
| **所有者（Owner）** | 控制技能分发的个人或组织 |
| **中转人（Relay）** | 将 Notes 从发送方转发给接收方的人类 |
| **Notes** | 包含下载指令和凭据的文字消息 |
| **注册中心（Registry）** | 存储技能元数据和认证信息的目录服务（不存储技能本身） |
| **认证（Certification）** | 技能已经过可信方审核的可验证证明 |
| **临时存储（Temporary Storage）** | 用于存储加密传输包的文件托管服务 |

## 4. 知识瓶子格式

### 4.1 当前格式（v1.x）

在 v1.x 中，知识瓶子是包含技能文件（通常包括 `SKILL.md`）的 `.tar.gz` 归档文件，使用 AES-256-CBC 加密。

### 4.2 结构化格式（v2.0 规划中）

在 v2.0 中，知识瓶子将在技能文件旁包含 `bottle.json` 清单。清单实现跨框架移植。

知识瓶子支持**三种表示类型**，反映 AI 技能的三大主要方法：

| 类型 | 描述 | 适用框架 |
|------|------|----------|
| **指令型（Instruction）** | LLM 阅读并执行的自然语言指令 | OpenClaw |
| **代码型（Code）** | 带类型化模式的可执行函数 | LangChain、AutoGen、CrewAI |
| **API 型** | HTTP 端点定义（OpenAPI） | Dify、Coze |

单个瓶子可以包含一种、两种或全部三种表示。接收框架选择它支持的模式。

> **注意：** `bottle.json` 扩展现有的 OpenClaw SKILL.md 格式。没有 `bottle.json` 的纯 SKILL.md 仍然是有效的 OpenClaw 技能。`bottle.json` 是跨框架移植的可选超集。

## 6. 协议流程 — 注册中心介导（v3.0 规划中）

未来版本将支持基于同意的注册中心介导传输：

1. **发现**：智能体 B 查询注册中心中可用的技能
2. **同意**：所有者明确批准（或设置自动审批策略）
3. **认证**：接收方根据注册中心验证技能的证书
4. **隐私**：技能在获得同意前始终留在所有者处

## 7. 传输包格式

### 7.1 技能打包

技能必须打包为以下格式之一：

| 类型 | 格式 | 使用场景 |
|------|------|----------|
| **目录** | `.tar.gz` | 多文件技能（代码 + 文档 + 配置） |
| **单文件** | 原始文件 | 单文件技能或文档 |

### 7.2 加密

所有传输包在上传前必须加密。

| 参数 | 要求 |
|------|------|
| **算法** | AES-256-CBC |
| **密钥派生** | PBKDF2 |
| **密码** | 至少 24 字符，随机生成，base64 编码 |
| **盐值** | 由 OpenSSL 自动生成 |

### 7.3 密码生成

密码必须：至少 24 字符长、使用密码学安全的随机源生成、一次性使用。

---

## 8. 临时存储

### 8.1 要求

临时存储服务必须：通过 HTTP 接受文件上传、返回可公开访问的下载 URL、在规定期限后自动删除文件、下载不需要认证。

### 8.2 推荐服务

| 服务 | 最大容量 | 过期时间 | 上传方式 |
|------|----------|----------|----------|
| litterbox.catbox.moe | 1 GB | 24小时 | `multipart/form-data` |
| c2cprotocol.org（规划中） | 待定 | 24小时 | C2C Registry API |

## 9. Notes 格式

Notes 是人类可读的传输工件。必须是纯文本，可被人类和 AI 智能体阅读。

### 9.1 结构

Notes 必须包含两个清晰分隔的区段：管理员区段（给人类中转人的摘要）和目标智能体区段（给接收方的机器可读指令）。

### 9.2 要求

- Notes 必须自包含
- Notes 必须包含解密密码和过期警告
- Notes 必须仅使用标准 shell 命令
- 人类中转人应能在不修改的情况下转发整个 Notes

---

## 10. 身份与认证（v3.0 规划中）

### 10.1 概述

注册中心提供身份和认证服务。在 c2cprotocol.org 上注册的技能携带关于其来源和审核状态的可验证元数据。

### 10.2 身份

**所有者**是控制技能分发的个人或组织。通过关联账号（如 GitHub、验证过的邮箱）或组织成员身份识别。

### 10.3 认证等级

| 等级 | 含义 |
|------|------|
| **自签名（self-signed）** | 所有者发布，无外部审核 |
| **已审核（reviewed）** | 另一个可信方已审核该技能 |
| **组织认证（org-certified）** | 组织已正式认证该技能 |

---

## 11. 同意与治理（v4.0 规划中）

### 11.1 同意模型

技能传授需要所有者的明确同意。同意可以是：

| 类型 | 描述 |
|------|------|
| **手动** | 所有者逐个审核并批准每个请求 |
| **基于策略** | 所有者定义规则（如"自动批准组织 X 的成员"） |
| **开放** | 所有者将技能标记为自由可用（等同于公开） |

### 11.2 组织治理

组织可以定义适用于其成员所有技能的策略：内部共享、合作伙伴访问、审计追踪。

## 12. 安全模型

### 12.1 威胁模型

C2C 假设：
- 智能体之间的通信信道**不可信**
- 临时存储**不可信**
- 人类中转人**可信**
- 注册中心（v3.0）**元数据可信**，存储不可信（技能从不存储在注册中心上）

### 12.2 安全属性

| 属性 | 机制 |
|------|------|
| **机密性** | AES-256-CBC 加密 |
| **完整性** | OpenSSL 验证解密 |
| **可用性** | 24小时过期限制暴露窗口 |
| **授权** | 人类中转（v1.x）或注册中心同意（v3.0+） |
| **不可重放** | 一次性密码；过期的存储链接 |
| **溯源** | 通过注册中心的认证链（v3.0+） |

### 12.3 人类参与审核保证

在 v1.x 中，安全性依赖于人类中转人。在 v3.0+ 中，人类监督从中转转向治理 — 所有者设置策略、审批请求并审计传输。

## 13. 传输要求

### 13.1 最低要求（v1.x 人类中继）

C2C 兼容的传输信道必须支持：至少 2,000 字符的文字消息、复制粘贴或消息转发。

### 13.2 兼容传输方式

任何文字信道均可工作：Telegram、微信、Discord、Signal、WhatsApp、邮件、短信，以及 c2cprotocol.org API（规划中）。

---

## 14. 实现要求

### 14.1 发送方要求

合规的发送方必须：打包、加密、上传、生成 Notes、呈现给人类（v1.x）或提交到注册中心（v3.0+）。

合规的发送方应该：包含 `bottle.json` 清单（v2.0+）。

### 14.2 接收方要求

合规的接收方必须：解析 Notes、下载、解密、解压、清理临时文件。

合规的接收方应该：根据注册中心验证认证（v3.0+）、从 `bottle.json` 选择适合其框架的表示（v2.0+）。

### 14.3 依赖（v1.x）

合规实现仅需：`openssl`、`curl`、`tar`、标准 shell 工具。无需额外库、SDK 或运行时。
