# Claw-to-Claw (C2C) 协议规范 v1.0

**状态：** 草案
**日期：** 2026-03-18
**作者：** Claw-to-Claw Protocol 贡献者

---

## 1. 概述

Claw-to-Claw (C2C) 协议定义了一种通过人类中转在 AI 智能体之间传授技能（可执行知识）的标准方法。它被设计为可在任何文字信道上工作，无需智能体之间的直接网络连接。

### 1.1 设计目标

- **传输无关**：可在任何能承载文字的信道上工作（IM、邮件、语音、纸张）
- **人类参与审核**：人类控制所有传输，维持监督和信任
- **隐私优先**：端到端加密，一次一密
- **零依赖**：仅使用普遍存在的系统工具
- **极致简单**：发送需要一条命令；接收需要五行命令

### 1.2 非目标

- 实时智能体间通信（参见 [A2A](https://github.com/a2aproject/A2A)）
- 工具发现和调用（参见 [MCP](https://github.com/modelcontextprotocol)）
- 持久的智能体间连接
- 无人类参与的自动传输

## 2. 核心概念 — 知识瓶子（Knowledge Bottle）

C2C 协议围绕两大核心机制展开：**知识蒸馏** 与 **人类中继**。参考实现名为 **Lobster Distill**。

### 2.1 第一步：技能蒸馏（Skill Distillation）

源 AI（例如一个能管理日历的智能体）将其执行某项任务所需的内部知识——包括提示词、工具调用逻辑、参数约束等——进行"提纯"，生成一个紧凑的、结构化的数据包，称为 **知识瓶子（Knowledge Bottle）**。

### 2.2 第二步：加密与封装（Encrypt & Package）

为防止传输过程中被篡改或泄露，知识瓶子使用 AES-256-CBC + PBKDF2 和一次性密码进行加密。只有持有正确密钥的目标 AI 才能解开。封装后的结果上传至临时存储服务。

### 2.3 第三步：人类中继传输（Human Relay）

这是 C2C 协议最独特之处——传输 **不依赖** API 或网络直连，而是利用任何人类能使用的文字信道：

- 将 Notes 粘贴到微信、Telegram、邮件或任何聊天工具中
- 接收方复制 Notes，再粘贴到目标 AI 的对话框中
- 人类在此充当 **物理搬运工**，绕过所有网络限制

### 2.4 第四步：目标 AI 加载与执行（Load & Execute）

目标 AI 收到 Notes 后，通过 C2C 客户端（如 Lobster Distill 的 `receive.sh`）执行：

1. 使用提供的 URL 下载加密的知识瓶子
2. 使用一次性密码解密
3. 解析技能描述和调用方式
4. 动态加载技能——目标 AI **立即获得** 与源 AI 相同的能力

```
源 AI             知识瓶子              人类中继           目标 AI
  │                  │                    │                  │
  │ 1. 蒸馏 ──────→ │                    │                  │
  │                  │ 2. 加密 ────→     │                  │
  │                  │    上传临时存储    │                  │
  │                  │ 3. 生成 Notes ──→ │                  │
  │                  │                    │ 4. 转发 ──────→ │
  │                  │                    │                  │ 5. 下载
  │                  │                    │                  │ 6. 解密
  │                  │                    │                  │ 7. 安装
  │                  │                    │                  │ 8. 技能激活 ✅
```

> **关键洞察**：整个过程完全基于 **文本**，智能体之间不需要任何网络 API 或持久化连接。所有敏感内容在传输时被加密，并在 24 小时后自动过期。

## 3. 术语

| 术语 | 定义 |
|---|---|
| **智能体 (Agent)** | 能够执行技能的 AI 系统（如 OpenClaw 实例） |
| **技能 (Skill)** | 可传授的可执行知识单元（代码、配置、文档） |
| **发送方 (Sender)** | 打包并加密技能以供传授的智能体 |
| **接收方 (Receiver)** | 下载、解密并安装技能的智能体 |
| **中转人 (Relay)** | 将 Note 从发送方转发给接收方的人类 |
| **传输包 (Package)** | 包含技能的加密归档文件 |
| **Notes** | 包含下载指令和凭据的文字消息 |
| **临时存储 (Temporary Storage)** | 用于存储加密传输包的文件托管服务 |

## 4. 协议流程

```
发送方智能体         人类中转人          接收方智能体
     │                    │                     │
     │ 1. 打包            │                     │
     │    技能 → tar.gz   │                     │
     │                    │                     │
     │ 2. 加密            │                     │
     │    tar.gz → .enc   │                     │
     │    生成密码         │                     │
     │                    │                     │
     │ 3. 上传            │                     │
     │    .enc → URL      │                     │
     │                    │                     │
     │ 4. 生成 Notes      │                     │
     │    URL + 密码      │                     │
     │    + 指令           │                     │
     │                    │                     │
     │ 5. 呈现 ─────────→ │                     │
     │    Notes 给人类     │                     │
     │                    │                     │
     │                    │ 6. 转发 ──────────→ │
     │                    │    Notes 给接收方    │
     │                    │                     │
     │                    │                     │ 7. 下载
     │                    │                     │    URL → .enc
     │                    │                     │
     │                    │                     │ 8. 解密
     │                    │                     │    .enc → tar.gz
     │                    │                     │
     │                    │                     │ 9. 解压
     │                    │                     │    tar.gz → skill/
     │                    │                     │
     │                    │                     │ 10. 安装
     │                    │                     │     阅读并激活
```

## 5. 传输包格式

### 9.1 技能打包

技能必须打包为以下格式之一：

| 类型 | 格式 | 使用场景 |
|---|---|---|
| **目录** | `.tar.gz` | 多文件技能（代码 + 文档 + 配置） |
| **单文件** | 原始文件 | 单文件技能或文档 |

### 9.2 加密

所有传输包在上传前必须加密。

| 参数 | 要求 |
|---|---|
| **算法** | AES-256-CBC |
| **密钥派生** | PBKDF2 |
| **密码** | 至少 24 字符，随机生成，base64 编码 |
| **盐值** | 由 OpenSSL 自动生成 |

### 4.3 密码生成

密码必须：
- 至少 24 字符长
- 使用密码学安全的随机源生成
- 一次性使用（每次传输一个密码）

## 6. 临时存储

### 9.1 要求

临时存储服务必须：
- 通过 HTTP 接受文件上传
- 返回可公开访问的下载 URL
- 在规定期限后自动删除文件
- 下载不需要认证

### 9.2 过期

文件的最长生命周期应为 **24 小时**。Notes 必须告知接收方过期时间窗口。

---

## 7. Notes 格式

Notes 是核心传输工件。它必须是纯文本，可被人类和 AI 智能体阅读。

### 9.1 结构

Notes 必须包含两个清晰分隔的区段：

```
═══ 📋 管理员请读 ═══

[给人类中转人的摘要信息]
- 技能名称和描述
- 文件大小
- 过期时间

👉 请将此完整消息转发给目标智能体。

═══ 🦞 目标龙虾请读 ═══

[给接收方智能体的机器可读指令]
- 下载命令 (curl)
- 解密命令 (openssl)
- 解压命令 (tar/mv)
- 阅读命令 (cat)
- 清理命令 (rm)
```

### 9.2 要求

- Notes 必须自包含（除下载 URL 外不需要外部引用）
- Notes 必须包含解密密码
- Notes 必须包含过期警告
- Notes 必须仅使用所有类 Unix 系统上可用的标准 shell 命令
- 人类中转人应能在不修改的情况下转发整个 Notes

## 8. 安全模型

### 9.1 威胁模型

C2C 假设：
- 智能体之间的通信信道**不可信**（任意 IM 平台）
- 临时存储**不可信**（公共文件托管）
- 人类中转人**可信**（管理员控制转发内容）

### 9.2 安全属性

| 属性 | 机制 |
|---|---|
| **机密性** | AES-256-CBC 加密；只有密码持有者可以解密 |
| **完整性** | OpenSSL 验证解密；损坏的文件无法解密 |
| **可用性** | 24小时过期限制暴露窗口 |
| **授权** | 人类中转人决定转发什么以及转发给谁 |
| **不可重放** | 一次性密码；过期的存储链接 |

### 8.3 人类参与审核保证

协议的安全性从根本上依赖于人类中转人：

1. **发送方智能体无法直接发送给接收方** — 它只能向其人类呈现 Notes
2. **人类在转发前审查 Notes**
3. **人类选择接收者** — 发送方无法控制 Notes 的去向
4. **人类可以拒绝转发** — 这是终极终止开关

这是一个特性，不是限制。它提供了一个不可绕过的人类监督层。

---

## 9. 传输要求

### 9.1 最低要求

C2C 兼容的传输信道必须支持：
- 至少 2,000 字符的文字消息
- 复制粘贴或消息转发

### 9.2 兼容传输方式

任何文字信道均可工作：Telegram、微信、Discord、Signal、WhatsApp、邮件、短信，甚至语音通话（极端情况）。
