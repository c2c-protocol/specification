# Claw-to-Claw (C2C) 协议规范 v2.0

**状态：** 草案
**日期：** 2026-03-23
**作者：** Claw-to-Claw 协议贡献者

---

## 1. 概述

Claw-to-Claw (C2C) 协议是一个**受治理的 AI 智能体互操作协议**。它定义了 AI 智能体（bot）如何建立信任、传授技能和进行协作——在每一步都有人类和组织的监督。

C2C 组织为**主协议 + 子协议**架构：

```
┌─────────────────────────────────────────────────────────┐
│                   C2C 协议 v2.0                          │
│                                                          │
│  ┌─────────────────┐    ┌───────────────────────────┐   │
│  │  子协议 1        │    │  子协议 2                  │   │
│  │  技能传授        │    │  可信协作                  │   │
│  │                 │    │                           │   │
│  │  "教我也会      │    │  "我们能一起做             │   │
│  │   这个技能"     │    │   这件事吗？"              │   │
│  └────────┬────────┘    └─────────────┬─────────────┘   │
│           │                           │                  │
│  ┌────────┴───────────────────────────┴─────────────┐   │
│  │               共享基础                            │   │
│  │                                                   │   │
│  │  身份与信任 │ 治理 │ 安全 │ 加密                    │   │
│  └───────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 1.1 C2C 解决的问题

| 问题 | 子协议 |
|------|--------|
| "这个智能体是否应该被允许学习这个技能？" | 技能传授 |
| "哪些可信的智能体可以跟我协作这个任务？" | 可信协作 |

### 1.2 设计目标

- **受治理的交互**：所有智能体交互都需要可验证的身份和信任
- **跨框架**：适用于各种 AI 框架（OpenClaw、LangChain、AutoGen、Dify 等）
- **人类参与**：人类控制信任关系并保持监督
- **去中心化决策**：公共节点提供可见性，不做决策
- **隐私优先**：端到端加密，默认私有
- **可渐进采用**：每个子协议和层级独立增值

### 1.3 非目标

- 实时智能体间通信（参见 [A2A](https://github.com/a2aproject/A2A)）
- 工具发现与运行时调用（参见 [MCP](https://github.com/modelcontextprotocol)）
- IDE-智能体会话管理（参见 [ACP](https://agentclientprotocol.com/)）
- 替代现有包注册中心（ClawHub、npm、pip）

### 1.4 C2C 与其他协议的关系

```
发现层：  A2A    — "你能做什么？"
学习层：  C2C    — "教我也会这个。"      （子协议 1）
协作层：  C2C    — "我们一起合作吧。"    （子协议 2）
执行层：  MCP    — "帮我做这件事。"
```

### 1.5 C2C 与包注册中心的区别

| | 包注册中心 | C2C (c2cprotocol.org) |
|---|---|---|
| **模型** | 仓库（文件托管在平台上） | 目录（仅元数据；技能保留在 owner 处） |
| **隐私** | 默认公开 | 默认私有 |
| **交付** | 任何人可下载 | Owner 必须同意 |
| **范围** | 特定框架 | 跨框架 |
| **类比** | Docker Hub | 亚马逊商城 |

# 第一部分：共享基础

## 2. 身份与信任

### 2.1 身份

C2C 协议中的每个 bot 都是一个**可验证实体**，而非匿名端点或临时昵称。

**Owner** 是控制 bot 参与 C2C 的个人或组织。Owner 通过以下方式标识：
- 关联账号（如 GitHub、已验证邮箱）
- 组织成员身份
- 加密密钥（未来）

### 2.2 信任关系

bot 之间的信任通过可验证的关系建立：

| 信任来源 | 描述 |
|---------|------|
| **介绍** | 受信任的第三方介绍两个 bot |
| **认证** | 权威机构为 bot 的身份或能力背书 |
| **公证** | 公共记录证明 bot 的属性 |
| **历史** | 反复成功的交互逐步建立信任 |

信任是**有方向的** — Bot A 信任 Bot B 不意味着 Bot B 信任 Bot A。

### 2.3 认证等级

| 等级 | 含义 |
|------|------|
| **自签名** | Owner 发布，无外部审核 |
| **已审核** | 另一受信方已审核 |
| **组织认证** | 组织已正式认证 |

### 2.4 公共发现节点

C2C 公共发现节点（如 c2cprotocol.org）为参与方提供基础可达性和发现能力。公共节点：

- **做**：维护已注册 bot 及其状态的最小可见性
- **做**：在 bot 之间转达消息和协作意图
- **不做**：代替任何参与方做决策
- **不做**：存储技能、知识或私有数据
- **不做**：充当调度器、分配器或推荐系统

> **核心原则：公共节点是传呼机，不是调度器。**

## 3. 治理

### 3.1 同意模型

C2C 中的所有交互都需要同意。同意可以是：

| 类型 | 描述 |
|------|------|
| **手动** | Owner 逐个审核并批准每个请求 |
| **基于策略** | Owner 定义规则（如"自动批准组织 X 的成员"） |
| **开放** | Owner 将资源标记为自由可用 |

### 3.2 组织治理

组织可以定义适用于其所有成员 bot 的策略：

- **内部共享**：组织内所有 bot 可自由交互
- **合作方访问**：预批准特定外部组织
- **审计追踪**：记录所有交互

---

## 4. 安全模型

### 4.1 威胁模型

C2C 假设：
- 智能体间的通信信道**不可信**（任意 IM 平台）
- 临时存储**不可信**（公共文件托管）
- 人类中转人**可信**（管理员控制转发内容）
- 公共发现节点**元数据可信**，不存储实际数据
- Bot 身份可通过第 2 节的信任机制**验证**

### 4.2 安全属性

| 属性 | 机制 |
|------|------|
| **机密性** | AES-256-CBC 加密 |
| **完整性** | OpenSSL 验证解密 |
| **可用性** | 24 小时过期限制暴露窗口 |
| **授权** | 人类中转（v1.x）或注册中心同意（v3.0+） |
| **不可重放** | 一次性密码；过期存储链接 |
| **来源可追溯** | 通过注册中心的认证链（v3.0+） |

### 4.3 加密标准

| 参数 | 要求 |
|------|------|
| **算法** | AES-256-CBC |
| **密钥派生** | PBKDF2 |
| **密码** | 至少 24 字符，随机生成，base64 编码 |
| **盐** | 由 OpenSSL 自动生成 |

## 5. 术语

| 术语 | 定义 |
|------|------|
| **Bot** | 能够参与 C2C 的 AI 智能体 |
| **Owner** | 控制 bot 参与 C2C 的个人或组织 |
| **技能** | 可传授的可执行知识单元 |
| **知识瓶子** | 经蒸馏和加密的 AI 技能包 — C2C 的核心传输产物 |
| **发送方** | 打包和加密技能进行传授的 bot |
| **接收方** | 下载、解密和安装技能的 bot |
| **中转人** | 将 Notes 从发送方转发给接收方的人类 |
| **Notes** | 包含下载指令和凭证的文本消息 |
| **注册中心 / 发现节点** | 存储 bot/技能元数据并提供发现的服务（不存储实际数据） |
| **认证** | 可验证的证明，表明某 bot 或技能已被受信方审核 |
| **可信候选集** | bot 本地维护的受信协作伙伴列表 |
| **协作意图** | 表达协作意愿的消息，通过公共节点转达 |
| **临时存储** | 用于存储加密包的文件托管服务 |

# 第二部分：子协议 1 — 技能传授

## 6. 核心概念 — 知识瓶子

技能传授子协议围绕两大核心机制：**知识蒸馏**和**传输**（通过人类中继或直接下载）。

### 6.1 第一步：技能蒸馏

源 bot 将执行某项任务所需的内部知识——包括提示词、工具调用逻辑、参数约束等——蒸馏为紧凑的结构化数据包，称为**知识瓶子**。

### 6.2 第二步：加密与封装

知识瓶子使用 AES-256-CBC + PBKDF2 和一次性密码加密。

### 6.3 第三步：传输

| 机制 | 描述 | 状态 |
|------|------|------|
| **人类中继** | 人类复制 Notes 通过任意文本渠道转发 | v1.0（当前） |
| **直接下载** | 接收方从 c2cprotocol.org 下载（需 owner 同意） | v3.0（规划中） |
| **API 传输** | 授权智能体间的程序化传输 | v4.0（规划中） |

### 6.4 第四步：加载与执行

目标 bot 解密知识瓶子，解析技能描述，动态加载技能——**立即获得**与源 bot 相同的能力。

---

## 7. 知识瓶子格式

### 7.1 当前格式（v1.x）

知识瓶子是 `.tar.gz` 归档，包含技能文件（通常包括 `SKILL.md`），使用 AES-256-CBC 加密。

### 7.2 结构化格式（v2.0 规划中）

v2.0 中，知识瓶子将包含 `bottle.json` 清单。支持三种表示类型：

| 类型 | 描述 | 框架 |
|------|------|------|
| **指令** | LLM 遵循的自然语言指令 | OpenClaw |
| **代码** | 带类型模式的可执行函数 | LangChain、AutoGen、CrewAI |
| **API** | HTTP 端点定义（OpenAPI） | Dify、Coze |

单个瓶子可包含一种、两种或全部三种表示。

## 10. 打包与 Notes 格式

### 10.1 技能打包

| 类型 | 格式 | 用途 |
|------|------|------|
| **目录** | `.tar.gz` | 多文件技能 |
| **单文件** | 原始文件 | 单文件技能 |

### 10.2 临时存储

临时存储服务必须：接受 HTTP 上传、返回公开下载 URL、定期自动删除、下载不需认证。

### 10.3 Notes 格式

Notes 必须是纯文本、自包含，包括：下载 URL、解密密码、Shell 命令、过期警告。

### 10.4 依赖（v1.x）

仅需：`openssl`、`curl`、`tar`、标准 shell 工具。

# 第三部分：子协议 2 — 可信协作

## 11. 概述

可信协作子协议定义了 bot 在已有可验证身份和信任关系的基础上，如何发现彼此的可用性、发起协作并建立工作关系——**不依赖中心化决策**。

本子协议**不**定义技能传授、知识交换或其他更高层能力。它只关注**建立可信协作**本身。

> **一句话：** 可信协作子协议用于支持 bot 在已有信任关系和公共发现底座之上，自主选择可信协作对象并发起协作，而公共节点仅提供可见性与传呼能力，不参与选择。

---

## 12. 前提条件

### 12.1 身份前提

每个 bot 都是可验证实体（第 2.1 节），而非匿名端点。

### 12.2 信任关系前提

bot 之间的可联系资格来自可验证的介绍、认证或公证关系（第 2.2 节）。

### 12.3 公共发现前提

互联网中存在 C2C 公共发现节点（第 2.4 节）。c2cprotocol.org 可作为保底入口。

---

## 13. 核心原则

### 13.1 协作先于开放连接

bot 之间不应默认向所有对象开放协作入口。协作发生在**预批准的候选集合**之内。

需求方首先关心的是：

> "我信任哪些 bot 可以参与这类协作"

而不是：

> "全网谁都可以来接这个请求"

### 13.2 公共节点只提供可见性，不提供决策

即使多个 bot 都具备相同能力，公共节点也**不应**替需求方决定由谁参与。

公共节点最多提供：

> "你认可的这些 bot 中，哪些当前可协作"

但不能提供：

> "我替你选了哪一个 bot 来协作"

### 13.3 选择权始终属于需求方

协作对象的最终选择权必须保留在需求方。公共节点帮助需求方看见候选对象的状态，但不应替代需求方的本地判断。

### 13.4 公共节点是传呼机，不是调度器

公共节点只做两件事：
1. 维护最小协作可见性
2. 转达协作意图

它不是：调度中心、仲裁者、交易所、推荐系统、统一派单平台。

---

## 14. 协作模型

### 14.1 第一步：形成可信协作候选集合

需求方依据已有的介绍、认证、公证或历史合作关系，在本地维护一个自己认可的 bot 集合。

### 14.2 第二步：查看协作可联系状态

需求方通过公共发现节点查看候选集合中哪些当前可联系、可响应、可参与。

公共节点提供的是**协作可见性**，而不是服务推荐。

### 14.3 第三步：需求方自主选择协作对象

如果多个候选 bot 都可参与，由需求方依据本地标准自行决定。

### 14.4 第四步：转达协作意图

公共节点将协作意图转达给目标 bot，使双方建立后续协作连接。

公共节点不决定协作内容，不参与协作判断，不代表任何一方做业务层决策。

## 15. 范围与边界

| 不负责 | 由谁负责 |
|--------|---------|

| 定义技能传授 | 子协议 1（第 6-10 节） |
| 定义全部业务协作语义 | 未来子协议 |

# 第四部分：传输与实现

## 16. 传输要求

### 16.1 最低要求（v1.x，技能传授）

C2C 兼容的传输信道必须支持：
- 至少 2,000 字符的文字消息
- 复制粘贴或消息转发

### 16.2 兼容传输方式

任何文字信道均可工作：Telegram、微信、Discord、飞书/Lark、Signal、WhatsApp、邮件、c2cprotocol.org API（规划中）。
